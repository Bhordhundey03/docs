---
title: "Create permanent virtual account"
description: "Generate a multi-use virtual account — either a balance-holding wallet or a pass-through collection account."
---

<Info>
  New to virtual accounts? Read the [overview](/api-reference/virtual-accounts/overview) first — it explains the difference between `permanent` and `permanent_collection`, which matters before you pick one.
</Info>

This endpoint creates a multi-use virtual account of one of two types:

- **`permanent`** — behaves like a wallet. Funds received into it accumulate as a balance you later move out via [Withdraw funds](/api-reference/funds/withdraw) or a [virtual account transfer](/api-reference/funds/virtual-account-transfer).
- **`permanent_collection`** — a pass-through account. It exists to identify incoming payments, not to hold them.

Both types can accept unlimited incoming payments and never expire on their own.

## Endpoint

```
POST /clients/business/virtual-accounts
```

## Headers

<ParamField header="x-api-key" type="string" required>
  Your public API key.
</ParamField>

<ParamField header="x-api-secret" type="string" required>
  Your API secret. Use a `stl_test_...` value in sandbox, `stl_live_...` in production.
</ParamField>

<ParamField header="businessId" type="string" required>
  Your business ID.
</ParamField>

## Request body

<ParamField body="firstName" type="string" required>
  First name to assign to the virtual account holder.
</ParamField>

<ParamField body="lastName" type="string" required>
  Last name to assign to the virtual account holder.
</ParamField>

<ParamField body="type" type="string" required>
  Account type. Accepted values: `"permanent"` (wallet — holds a balance) or `"permanent_collection"` (pass-through — see [overview](/api-reference/virtual-accounts/overview)).
</ParamField>

<ParamField body="bvn" type="string" required>
  The account holder's Bank Verification Number. Must be exactly 11 digits.
</ParamField>

<ParamField body="reference" type="string" required>
  A unique reference string you generate to identify this account in your system. Must be unique across your account — creating a second account with a reference you've already used returns an error.
</ParamField>

## Response

Returns `201 Created` on success.

<ResponseField name="status" type="boolean">
  `true` on success.
</ResponseField>

<ResponseField name="message" type="string">
  Confirmation message, e.g. `"Account generated successfully"`.
</ResponseField>

<ResponseField name="data.accountNumber" type="string">
  The NUBAN account number generated for this virtual account.
</ResponseField>

<ResponseField name="data.accountName" type="string">
  The name on the account, derived from the `firstName` and `lastName` provided.
</ResponseField>

<ResponseField name="data.clientRef" type="string">
  The `reference` you supplied, echoed back for confirmation.
</ResponseField>

<ResponseField name="data.systemRef" type="string">
  A unique reference generated by Interstellas for this account.
</ResponseField>

## Errors

| Status | Cause |
|---|---|
| `400 Bad Request` | `reference`, `firstName`, `lastName`, or `bvn` missing |
| `400 Bad Request` | `bvn` is not exactly 11 characters |
| `400 Bad Request` | `type` is anything other than `"permanent"` or `"permanent_collection"` |
| `400 Bad Request` | `reference` already exists (`"Reference already exists"`) |
| `400 Bad Request` | `"No payment account for this business"` — set up your business's own payment account first |
| `401 Unauthorized` | `businessId` header missing, or credentials invalid |

See [Errors](/essentials/errors) for the full envelope and error code reference.
